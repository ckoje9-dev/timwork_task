## 데이터 분석

### metadata.json을 어떻게 해석했나요?

크게 **프로젝트 정보**, **공종 목록**, **도면 트리** 세 영역으로 구분됩니다.

도면 트리에서 주목한 점은 **위치와 커버리지를 polygon 좌표로 표현**한 것입니다. 상위 도면에서 이 건물이 어디에 위치하는지(`position.vertices`), 해당 공종이 도면 내에서 어떤 영역을 다루는지(`polygon.vertices`), 리비전마다 변경 구역이 어디인지(`revision.polygon.vertices`) — 이 세 가지 polygon 데이터가 단순한 "파일 목록" 이상의 공간적 맥락을 담고 있습니다.

`imageTransform`은 서로 다른 해상도·비율의 이미지를 하나의 기준 이미지 위에 정렬하기 위한 파라미터입니다. `polygonTransform`은 polygon의 global 좌표를 특정 이미지 픽셀 공간으로 변환하는 파라미터로, 두 transform의 역할이 다릅니다.

**특수 케이스 정리:**

| 케이스 | 도면 | 해석 |
|--------|------|------|
| Region 분할 | 101동 구조 | 대형 도면을 A/B 구역으로 나눠 각자 독립적인 리비전 관리. 실무에서 흔한 패턴. |
| 리비전마다 polygon이 다름 | 주민공동시설 건축 | 설계 변경 범위가 리비전마다 달라지는 케이스. polygon이 discipline 레벨이 아닌 revision 레벨에 있음. |
| polygon 없는 공종 | 주차장 구조 | 기준 도면 자체여서 관심 영역 지정 불필요. imageTransform만 존재. |

### 더 나은 데이터 표현 방법이 있을까요?

현재 구조는 **도면 중심(bottom-up)** 입니다. 각 도면 아래 공종이 매달리고, 그 아래 리비전이 매달리는 구조여서 "건축 공종의 모든 도면"을 조회하려면 전체 트리를 순회해야 합니다.

실무 도면 관리 방식은 오히려 **공종이 먼저(top-down)** 입니다. 설계 업체가 공종별로 다르고, 도면 체계도 공종별로 독립적으로 운용됩니다. 공종을 1차 키로 하고, 그 아래 도면을 찾아가는 방식이 탐색 성능과 실무 워크플로 모두에서 더 자연스럽습니다.

추가로, 현재 도면 식별자가 단순 숫자(`01`, `09` 등)인데 실제 도각(도면번호체계 — 예: `A-1-101`, `S-1-101` 등)을 그대로 사용하면 도면을 찾는 담당자가 코드 없이도 의미를 읽을 수 있습니다.

## 접근 방식

### 전체 과제를 어떤 순서와 방식으로 해결하셨나요?

1. **사용자 시나리오 분석** — 김철수 씨의 3가지 태스크(최신 현황 파악, 공종 간 간섭 확인, 변경 이력 추적)를 기능 요구사항으로 분해
2. **데이터 구조 파악** — `metadata.json`을 분석하여 탐색 축 결정 (공종 우선), 특수 케이스(region, 리비전별 polygon) 처리 방식 설계
3. **레이아웃 설계** — 2패널(트리 + 뷰어) 확정, 리비전 패널은 상단 드롭다운으로 통합
4. **기술 스택 선택** — 보일러플레이트 최소화(Zustand), 빠른 프로토타이핑(Tailwind), 백엔드 협업 준비(Mock API 레이어)
5. **핵심 기능 먼저** — 도면 표시 → 트리 탐색 → 리비전 전환 순으로 필수 기능 완성 후 비교 모드·폴리곤 탐색 등 부가 기능 추가
6. **Mock API 레이어** — 백엔드 없이 개발하되, 실제 API 전환 시 한 줄 변경으로 스위칭 가능하도록 설계

## UI 설계 결정

### 어떤 레이아웃을 선택했고, 왜 그 방식이 최선이라고 판단했나요?

```
┌─────────────┬──────────────────────────────────────┐
│  공종 트리   │  [브레드크럼 / 리비전 피커]  [주석·비교] │
│  (좌, 288px) ├──────────────────────────────────────┤
│             │                                      │
│             │         도면 뷰어 (줌/패닝)            │
│             │                                      │
└─────────────┴──────────────────────────────────────┘
```

2패널 구조 — 탐색(트리)과 확인(뷰어)을 한 화면에서 전환 없이 처리합니다. 도면을 보면서 트리에서 다른 도면으로 바로 넘어가는 흐름이 자연스럽습니다.

리비전 패널은 처음에 3패널(트리·뷰어·리비전 이력)로 검토했으나, 리비전 수가 많지 않고 드롭다운 선택이 더 직관적이어서 상단 헤더 드롭다운으로 통합했습니다. 뷰어 공간을 최대한 확보하는 것이 도면 작업에 더 중요합니다.

### 고려했던 대안들과 각각의 장단점은 무엇이었나요?

**탐색 구조: 공종 우선 vs 공간 우선**

| 방식 | 장점 | 단점 |
|------|------|------|
| 공간 우선 (배치도 클릭) | 직관적인 지도 탐색, "어디"에 대한 질문에 강함 | 공종 횡단 탐색 불편, 배치도 없는 환경에서 불가 |
| **공종 우선 (트리) — 선택** | 실무 워크플로 일치, "건축 도면 전부" 조회 용이 | 공간 컨텍스트 약함 |

공간 탐색은 폴리곤 네비게이션으로 보완했습니다 — 전체 배치도·101동 평면도에서 영역을 클릭해 해당 도면으로 이동합니다.

**리비전 비교: 슬라이더 vs 오버레이 레이어**

| 방식 | 장점 | 단점 |
|------|------|------|
| Before/After 슬라이더 | 두 버전의 차이가 명확 | 두 리비전만 비교 가능 |
| **색상 오버레이 레이어 — 선택** | REV1~REV3 동시 비교 가능, 공종 간 간섭도 확인 | 도면이 복잡하면 가독성 저하 |

시나리오 3 ("REV1→REV3 변경 방향 논의")에는 여러 리비전을 동시에 켜고 끄며 볼 수 있는 레이어 방식이 적합합니다. 각 레이어의 투명도와 가시성을 개별 조절할 수 있고, 비교 모드 진입 시 폴리곤 오버레이를 자동으로 숨겨 도면 가독성을 확보합니다.

## 어려웠던 점 및 개선 방안

### 어려웠던 점

**1. 좌표계 변환 수식 도출**

`polygon.vertices`의 global 좌표를 revision 이미지 픽셀 공간으로 변환하려면 `imageTransform`의 `rotation`을 역방향으로 적용해야 합니다.

```
dx = gx - tx.x
dy = gy - tx.y
imgPx = (dx * cos(-θ) + dy * sin(-θ)) / tx.scale + imageWidth / 2
imgPy = (-dx * sin(-θ) + dy * cos(-θ)) / tx.scale + imageHeight / 2
```

`rotation`이 이미지를 "global 방향으로 돌리는" 각도이므로 polygon을 이미지 공간으로 가져오려면 반대 방향(-θ)으로 역회전해야 한다는 점을 확인하는 데 시간이 걸렸습니다.

**2. 비교 모드 오버레이 스케일 공식**

건축 JPEG(1,232px, scale ≈ 4.02)와 구조 PNG(4,962px, scale = 1)처럼 해상도가 다른 이미지를 같은 물리 영역에 겹치려면 CSS transform scale을 올바르게 계산해야 합니다.

처음에 `displayScale = baseTx.scale / overlayTx.scale`로 구현했는데, 이 방향이 반전되어 구조 도면이 4배 크게 렌더링되었습니다. 올바른 공식은 다음과 같습니다:

```
displayScale = overlayTx.scale / baseTx.scale   // 0.248 → 오버레이를 축소
lx = (overlayTx.x - baseTx.x) / baseTx.scale   // 중심 오프셋 보정
```

`scale` 필드의 의미가 "오버레이 1px당 기준 이미지 픽셀 수"임을 정확히 파악한 후 해결했습니다.

**3. Set/Map 직렬화**

Zustand 상태로 `Set<string>`, `Map<string, T>` 사용 시 `JSON.stringify`가 빈 객체(`{}`)를 반환하여 localStorage 영속이 되지 않는 문제. 저장 시 배열로 직렬화하고, 초기화 시 복원하는 패턴으로 해결했습니다.

**4. 스토어 간 순환 참조**

이슈 삭제 시 연관 핀도 제거해야 하는데, `issue.store`가 `drawing.store`를 import하거나 반대 방향으로 import하면 순환 참조가 발생합니다. 두 스토어가 서로를 직접 참조하지 않고, `IssueDetailModal` 컴포넌트에서 두 스토어를 모두 사용하는 방식으로 해결했습니다.

## 개선 포인트 및 미완성 항목

### ㄱ. 도면 비교 정렬(Align) 정확도

현재는 `imageTransform`의 픽셀 좌표로 스케일을 맞춰 오버레이를 정렬합니다. 그러나 도면마다 프린터 설정(여백, 해상도, 용지 비율)이 조금씩 다르면 픽셀 기준 align으로는 구조적으로 정확히 맞추는 데 한계가 있습니다.

**근본적인 해결 방법**: 이미지 픽셀 좌표 대신 CAD 파일의 XY 좌표계를 직접 활용해야 합니다. CAD 원본 파일에서 기준점(설계 원점, 측량 기준점 등)을 추출하면 어떤 출력 설정에서도 정확한 align이 가능합니다. 또한 도면 내 기준점 심볼(십자 마크 등)을 자동 인식하거나 사용자가 수동 지정할 수 있는 UI도 보완책이 될 수 있습니다.

### ㄴ. 도면 탐색 프로세스와 polygon 좌표 추출

현재 metadata의 polygon 좌표는 각 도면마다 수작업 또는 별도 툴로 추출해야 합니다. 건물마다 형태가 다르기 때문에, 실제 서비스화하려면 polygon 좌표를 추출·관리하는 프로세스가 필요합니다.

실무 도면 세트에는 탐색을 위한 주석 객체가 이미 체계적으로 사용됩니다:
- **단면선 기호(Section Mark)**: 단면도가 어디서 절단되었는지 위치 표시
- **콜아웃 심볼(Callout)**: 상세도나 확대도를 참조하는 심볼
- **리비전 클라우드(Revision Cloud)**: 변경된 영역을 구름 모양으로 표시

이 주석 객체들을 CAD/PDF에서 파싱하여 탐색에 활용하는 방식을 주기능으로 하되, 현재의 위치 인식 기반 폴리곤 탐색은 보조 수단으로 병행하는 것이 이상적입니다.

### ㄷ. 도면 트리 구조 재검토

현재 트리는 "전체 배치도 → 101동 → 건축" 순서의 공간 기반 계층입니다. 직관적이고 시각적이지만, 실무에서는 다음과 같은 문제가 생깁니다:

- 공종별로 설계 업체가 달라, 건축·구조·설비·소방 담당자는 자기 공종 도면만 관리합니다.
- 설계 변경·수정 업무 시 공종별 체계가 우선이며, 위치 기반 분류는 부차적입니다.
- 건물 배치가 변경되면 트리 계층 자체가 흔들리는 구조적 취약점이 있습니다.

공종을 1차 키로 하고, 그 아래 도면을 찾아가는 top-down 방식으로 트리를 재설계하면 실무 도면 관리 흐름과 더 잘 맞습니다. 위치 기반 탐색은 폴리곤 네비게이션으로 보완합니다.

### ㄹ. 프로젝트 CRUD 미완성

현재는 단일 프로젝트 고정 화면입니다. 실제 서비스에서는 한 계정이 여러 현장·프로젝트를 관리해야 합니다:
- 프로젝트 생성·수정·삭제
- 프로젝트 선택 후 해당 도면·이슈 컨텍스트로 전환
- 프로젝트별 metadata.json 관리

### ㅁ. 사진대지·멤버·보안 기능 미완성

**사진대지**: 현장 사진을 도면 위 좌표에 등록하고, 사진대지 문서(표 형식의 사진 보고서)를 생성·다운로드하는 기능. 이슈 핀과 유사한 핀 기반 접근 방식으로 구현할 수 있습니다.

**멤버·권한 관리**: 프로젝트별 멤버 등록·삭제, 역할(관리자·편집자·뷰어) 기반 접근 제어. 공종별 담당자를 지정하면 이슈 생성 시 담당자 자동 추천도 가능합니다.

### ㅂ. 이슈 기능 보완

현재 이슈와 도면의 연결은 단방향에 가깝습니다. 다음 개선이 사용성을 높일 것입니다:

- **이슈 → 도면 이동**: 이슈 상세에서 참조 도면을 클릭하면 해당 도면 + 핀 위치로 직접 이동
- **이슈 댓글**: 협력업체·담당자 간 이슈별 토론 스레드
- **이슈 간 관계**: 선행 이슈, 중복 이슈, 관련 이슈 연결 (예: 간섭 이슈 → 수정 이슈 → 확인 이슈 흐름)
- **담당자 지정 프로세스**: 계정 기능 탑재 후 이슈 생성 시 담당자 검색·지정, 알림 발송

## 상태 관리 설계

### Zustand 스토어 구조

```
drawing.store.ts  — 도면 선택, 레이어, 비교 모드, 이슈 핀, 북마크, 주석 표시

```

### localStorage 영속성

세 스토어 모두 localStorage를 사용해 새로고침 후에도 상태를 복원합니다.

| localStorage 키 | 저장 내용 |
|----------------|----------|
| `timwork-drawing-bookmarks` | `string[]` — `${drawingId}-${discipline}` 배열 |
| `timwork-issue-bookmarks` | `[string, BookmarkedIssueInfo][]` — Map entries 직렬화 |

### 이슈 핀 연동 구조

`IssuePin.issueId`로 도면 핀과 이슈를 양방향 연결합니다:

- **핀 배치 → 이슈 생성**: `addIssuePin` 후 이슈 생성 모달에서 `updateIssuePinData`로 이슈 ID를 핀에 기록
- **이슈 삭제**: `issuePins.filter(p => p.issueId === issue.id)`로 연관 핀을 찾아 자동 제거
- **순환 참조 방지**: `issue.store`와 `drawing.store`가 서로를 직접 import하지 않고, `IssueDetailModal` 컴포넌트에서 두 스토어를 함께 사용

## 폴리곤 좌표계 설계

### global 좌표 → 이미지 픽셀 변환

```
dx = gx - tx.x
dy = gy - tx.y
imgPx = (dx * cos(θ) + dy * sin(θ)) / tx.scale + imageWidth / 2
imgPy = (-dx * sin(θ) + dy * cos(θ)) / tx.scale + imageHeight / 2
```

`imageTransform`의 `rotation`이 이미지를 global 방향으로 돌리는 각도이므로, polygon을 이미지 픽셀 공간으로 가져오려면 반대 방향(-θ)을 적용합니다. 수식에서 cos(-θ) = cos(θ), sin(-θ) = -sin(θ)를 전개한 형태입니다.

### 폴리곤 오버라이드 (`src/config/polygonOverrides.ts`)

metadata.json 수정 없이 특정 리비전의 폴리곤을 UI 레벨에서 교체합니다:

```ts
// null → 폴리곤 숨김 / 배열 → 커스텀 vertices 사용
export const revisionPolygonOverrides: Record<string, Record<string, PolygonOverride>>
```

주민공동시설(09) 건축 도면 적용 예:
- REV1: `null` (초기 설계 — 변경 범위 없음)
- REV2/REV3: 실제 변경 구역을 나타내는 커스텀 사각형 좌표

### 도면 비교 오버레이 스케일 공식

```
displayScale = overlayTx.scale / baseTx.scale
lx = (overlayTx.x - baseTx.x) / baseTx.scale
ly = (overlayTx.y - baseTx.y) / baseTx.scale
```

`scale` 필드는 "기준 이미지 픽셀 per 해당 이미지 픽셀"입니다. 건축(1,232px, scale ≈ 4.02)과 구조(4,962px, scale = 1)의 경우 `displayScale = 1 / 4.02 ≈ 0.248`이 되어 구조 PNG를 0.248배 축소하면 4962 × 0.248 ≈ 1,231px로 건축 이미지와 동일한 물리 크기가 됩니다.

## Mock API 설계

각 API 함수는 `apiConfig.useMock` 플래그로 Mock/Real을 분기합니다:

```ts
export async function getIssues(params): Promise<Issue[]> {
  if (apiConfig.useMock) return mockDelay(_getIssues(params));
  return apiFetch<Issue[]>('/issues', { ... });
}
```

| 기능 | Mock 구현 | 실제 API 경로 |
|------|----------|-------------|
| 도면 트리 | `/metadata.json` fetch + 가공 | `GET /drawings/tree` |
| 이슈 CRUD | `MOCK_ISSUES` 배열 인메모리 | `GET/POST/PUT/DELETE /issues` |
| 이슈 통계 | MOCK_ISSUES 집계 | `GET /issues/stats` |
| 도면 업로드 | drawing.store 직접 처리 (blob URL) | `POST /drawings` |
| 이슈 핀 | drawing.store 인메모리 | 백엔드 협의 필요 |
